# 管线演进史

从 130 行硬编码到完整的视频生产线——每个版本解决了什么、放弃了什么、为什么。

---

## v1 — 验证可行性（`create_video.py`）

### 做了什么

130 行代码，所有内容硬编码在 Python 里。旁白文字、图片路径、字幕文字都是写死的。

```python
VOICEOVER_TEXT = """
距离地球6500光年，这里是创生之柱。...
"""
IMAGES = [
    {"path": "F:/tmp/pillars_wide.png", "text": "📍 距离地球6500光年"},
    ...
]
```

配音用 `edge_tts` 一次生成整段，图片均分时间轮播，Python 写了个简陋的缩放函数。

### 为什么这样开始

验证一个核心假设：**AI 图片 + TTS 配音 + MoviePy = 能看的竖屏视频**。答案是：能。

### 放弃了什么

- 配音时长与图片不匹配（一段音频对 4 张图，各自分离）
- 缩放几乎看不出来
- 改内容 = 改代码

---

## v2 — 工程化（`video_pipeline_v2.py`）

### 核心创新：脚本与代码分离

不再把内容写死在 Python 里，引入 JSON 脚本：

```
内容创作者 → 写 JSON  → 管线读取 JSON → 生成视频
            （不用懂代码）
```

### 配音升级

v1 是一段 TTS 对多张图，v2 改为**每场景独立配音**：

```json
{
  "id": 3,
  "narration": "一颗巨齿鲨的牙齿有17厘米长...",
  "rate": "+3%",
  "pitch": "+2Hz",
  "volume": "+0%"
}
```

三个参数控制每场景的情感：
- `rate` 语速：悬念时减速、兴奋时加速
- `pitch` 音高：恐惧低沉、激昂升高
- `volume` 音量：强调加大

还解决了长文本 TTS 返回空音频的问题——按标点拆成短句、分段合成、再拼接。

### 运镜

MoviePy 的 `Resize + Crop`，两档：`zoom_in_slow`（1.08x）和 `wide`（静态）。

### 视频结构

```
标题卡（大字叠加在背景图上）
    ↓
场景1（图 + 配音 + 底部字幕 + 画面标注）
    ↓
场景2
    ↓
  ...
    ↓
结束卡（"下期想看什么？评论区见"）
```

### 放弃了什么

- 字幕干扰二次元画风——文字让画面不干净
- MoviePy 缩放不够流畅——看起来"抽搐"
- 结构太固定——不能没有标题卡和结束卡

---

## v3 — 质量顶峰（`video_pipeline_v3.py`）

### 核心理念转变

> **让画面说话。** 旁白足够讲故事，不需要字幕。

删掉了 v2 的所有文字叠加（标题卡、字幕、画面标注、结束卡），只剩图片和声音。

### FFmpeg zoompan

用 FFmpeg 替代 MoviePy 做运镜。FFmpeg 的 `zoompan` 滤镜是逐帧渲染的，效果流畅得多。

6 种运镜方向：

```
in    → 逐渐推进（揭示真相）
out   → 逐渐拉远（宏大收尾）
left  → 向左平移
right → 向右平移
up    → 向上平移（仰望）
down  → 向下平移（深海坠落）
```

每个场景自动轮换方向，8 个场景跑完 `["in","right","out","left","up","in","left","out"]`，不重样。

### 配音优化

- MD5 缓存：相同旁白文本不再重复请求 edge_tts
- 重试从 3 次提到 5 次
- 文件名用 hash 去重，避免同一段话多个场景冲突

### 结构简化

```
开场（第一张图 Fade In, 2s）
场景1..N（图 + 配音 + zoompan，每场景 -0.5s 交叉过渡）
结尾（最后一张图 Fade Out, 3s）
```

开场和结尾共用首尾图片，不再需要专门的标题/结束卡。

## 每一步"为什么"

### 为什么 v1→v2：内容和代码必须分离

v1 做完创生之柱后，想做巨齿鲨——发现要改代码。这意味着每次做新视频都要写 Python。不是可持续的工作流。

JSON 脚本解决了这个问题：内容创作者只需要懂 JSON 格式，不用碰代码。

### 为什么 v2→v3：画面质量比功能多更重要

v2 有了字幕、标注、情感控制——"功能齐全"。但出来的视频看起来像 PPT，不像短视频。

删掉字幕不是因为懒，是因为：
1. 二次元画风的氛围感是核心，文字破坏沉浸
2. 科普短视频的核心是让观众"想看下去"，不是"读下去"
3. 旁白已经把故事讲清楚了，字幕是冗余信息

### 为什么用 FFmpeg 而不是 MoviePy 做运镜

MoviePy 做缩放是在 Python 里逐帧计算，开销大且不够平滑。FFmpeg 的 `zoompan` 滤镜是 C 实现，逐帧插值，效果是原生级的。

代价是增加了对 FFmpeg 外部工具的依赖——但 FFmpeg 几乎每个人都该装，这个代价值得。
